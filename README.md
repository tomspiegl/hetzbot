# hetzbot

Agent-managed hosting framework for Hetzner Cloud. Declarative infra
(OpenTofu), Tailscale-only access, encrypted off-host backups (restic),
stateful third-party services as compose stacks (Postgres today), your
own services as systemd units cloned from GitHub.

You clone this repo, scaffold a **fleet** repo beside it, tell an
agent (Claude Code) what you want — it handles provisioning, deploys,
backups, reviews, and teardown via a set of documented skills.

## What you get

- **Provisioning** via OpenTofu — one entry in `hosts.tfvars` → one
  Hetzner VM + firewall + optional DNS record.
- **First-boot hardening** via cloud-init — default-deny ufw, Tailscale
  SSH only, `unattended-upgrades`, hardened sshd.
- **On-demand infra skills** — Docker, restic, Caddy, Postgres,
  Node / Python install only when a service needs them. Nothing
  baked into the base image.
- **Safe-by-default services** — systemd hardening drop-in, per-service
  `/srv/<svc>/.env` with generated passwords, lockfile-required deploys,
  `/etc/npmrc` supply-chain lockdown.
- **Backups** — restic (encrypted, off-host) + weekly Hetzner Backups
  (whole-disk rollback) + optional manual snapshots for pre-change
  safety.
- **Severity-tagged audit** — `review-host` orchestrator runs every
  skill's own `review.sh` and aggregates findings.

## How it works

Two repos, sibling-cloned on your laptop:

```
~/Develop/
├── hetzbot/           this repo — framework (skills/, docs/, templates/)
└── <your-fleet>/      generated by init-fleet — tofu/, hosts.tfvars, services/, .env
```

The fleet's `.env` sets `HETZBOT_ROOT=../hetzbot`. All scripts resolve
framework vs. fleet paths at runtime — so hetzbot is reusable across
any number of fleets without copying code.

The agent runs on your laptop and inherits your Tailscale identity.
When it calls `ssh <host>`, it goes over the tailnet as you; no SSH
keys on hosts, no public SSH port.

Skills are small directories under `skills/<group>/<skill>/` containing
a `SKILL.md` playbook plus optional `install.sh`, `review.sh`,
`backup.sh`. Four groups:

| Group | What |
|---|---|
| `hetzner/` | Cloud lifecycle (init-fleet, add-host, remove-host, snapshot-host, check-fleet, review-host) |
| `ops/` | Service lifecycle verbs (add-service, remove-service, deploy, restore) |
| `infra/` | Installable daemons / stacks (docker, restic, caddy, postgres, pi optional) |
| `runtimes/` | Language runtimes installed on demand (node, python/uv) |

## What you need on your laptop

- **A coding agent / LLM.** hetzbot's skills are written as agent
  playbooks — they're executable by an LLM that can read SKILL.md
  files, run shell commands, and edit files. The reference setup is
  [Claude Code](https://claude.ai/code); equivalent agents work if
  they honor the same SKILL.md convention.
- **CLI tools:** `opentofu`, `tailscale`, `just`, `git`, `openssl`,
  `jq`, `curl`, `rsync`. On macOS:
  ```bash
  brew install opentofu tailscale just git openssl jq
  ```
- **Tailscale signed in** with your work identity
  (`tailscale up`), so the agent can SSH to hosts as you.
- **A personal vault** (1Password, Bitwarden, macOS Keychain — whatever
  you use) holding the four fleet credentials. See
  [docs/quickstart.md](docs/quickstart.md) for the exact list.

Without the LLM you can still read the SKILL.md files as documentation,
but the speed comes from handing the repo to an agent.

## Quickstart

**1. Clone this repo beside where your fleet will live:**

```bash
mkdir -p ~/Develop && cd ~/Develop
git clone https://github.com/tomspiegl/hetzbot
```

**2. Open your coding agent in the same directory:**

```bash
claude   # Claude Code, or equivalent
```

**3. Paste this prompt to get the guided bootstrap:**

```
I'd like to set up a new hetzbot fleet named `my-fleet` at
`~/my-fleet`. First walk me through the external prerequisites
(Hetzner account + project + API token, Object Storage bucket with
S3 keys, Tailscale tailnet, DNS zone if I'll serve HTTPS, personal
vault with the four credential items). Then run the `init-fleet`
skill to scaffold the fleet repo. Finally, guide me through adding
my first host with the `add-host` skill.
```

The agent will read `CLAUDE.md` (this repo's agent guide), consult the
relevant SKILL.md files, and walk you step by step — asking one
question at a time, never skipping a confirmation.

## Non-goals

HA, cross-host DB, Kubernetes, CI pipelines, public SSH, plain HTTP,
untrusted multi-tenant workloads. See
[docs/non-goals.md](docs/non-goals.md) for the full list and each
named upgrade path.

## Status

Framework is feature-complete for the single-operator / few-operator
setup it targets. Anything beyond the non-goals is out of scope by
design.
